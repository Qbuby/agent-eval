# Agent 自动化测试优化闭环系统 — 项目架构与方案

## 1. 项目概述

本系统是一个针对 LangChain/LangGraph Agent 的**全链路自动化测试优化闭环平台**。核心能力：

- **全链路 Trace 采集**：通过 LangChain Callback 机制零侵入采集工具调用序列、推理链路、最终输出
- **多维度自动评估**：五维评分体系（输出正确性、工具序列、推理质量、性能、错误恢复），支持规则 + LLM-as-Judge 混合评分
- **自动优化闭环**：失败模式分析 → 策略生成 → 策略应用 → 重新评估，直到指标达标或触发安全阀

技术选型：Python 3.11+ / LangChain / SQLAlchemy Async / PostgreSQL / Grafana

## 2. 整体架构

### 2.1 三层架构 + 闭环控制器

```
┌─────────────────────────────────────────────────────────────────┐
│                    闭环控制器 (LoopController)                    │
│           收敛判断 · 停滞检测 · 回归保护 · A/B 分流                │
└──────┬──────────────────┬──────────────────┬────────────────────┘
       │                  │                  │
       ▼                  ▼                  ▼
┌──────────────┐  ┌───────────────┐  ┌───────────────────┐
│   数据层      │  │   评估层       │  │    优化层          │
│              │  │               │  │                   │
│ TestCase     │→│ TraceCollector │→│ FailureAnalyzer   │
│ DatasetMgr   │  │ DimensionScorer│  │ StrategyGenerator │
│ CaseGenerator│  │ RegressionDet. │  │ StrategyApplicator│
└──────────────┘  └───────────────┘  └───────────────────┘
       │                  │                  │
       └──────────────────┴──────────────────┘
                          │
                   ┌──────▼──────┐
                   │ PostgreSQL  │──→ Grafana Dashboard
                   └─────────────┘
```

### 2.2 闭环控制流

```
加载用例 → 执行 Agent → 采集 Trace → 多维评分
    ↑                                    │
    │                              达标? ──→ 是 → 输出报告
    │                                │
    │                                否
    │                                ▼
    └── 补充用例 ← 失败分析 → 生成策略 → 应用策略 → 重新测试
```

每轮迭代的详细流程：

1. 从数据层加载测试用例集
2. 创建 Agent 实例，挂载 TraceCollectorCallback
3. 并发执行所有用例，采集完整 Trace
4. 五维评分器分别打分，计算加权总分
5. **收敛判断**：总分 >= 目标分数 → 结束，输出最终报告
6. **回归检测**：对比 baseline，发现退化 → 回滚到最佳配置
7. **停滞检测**：连续 N 轮改进不足 → 安全停止
8. **失败分析**：LLM 分类失败原因，按类别聚类
9. **策略生成**：LLM 基于失败聚类生成优化策略（prompt/工具/参数调整）
10. **策略应用**：将策略应用到 Agent 配置副本
11. 回到步骤 1，进入下一轮

## 4. 数据层详细设计

### 4.1 核心数据模型

**TestCase** — 测试用例的完整定义：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识 |
| `dataset_version` | str | 所属数据集版本 (如 "v1.0.0") |
| `name` / `description` | str | 用例名称和描述 |
| `tags` | list[str] | 标签，支持 GIN 索引查询 |
| `source` | enum | "manual" / "auto_generated" / "failure_derived" |
| `input_messages` | list[dict] | Agent 输入消息 |
| `expected_output` | str | 预期输出（精确/语义匹配参考） |
| `expected_output_criteria` | list[str] | 自然语言验收标准 |
| `expected_tool_calls` | list[ToolCallExpectation] | 预期工具调用序列 |
| `max_tool_calls` / `max_latency_ms` / `max_tokens` | int | 性能阈值 |
| `eval_weights` | EvalWeights | 五维评估权重 |
| `scoring_mode` | enum | "rule" / "llm" / "hybrid" |

**ToolCallExpectation** — 预期的单次工具调用：

| 字段 | 说明 |
|------|------|
| `tool_name` | 工具名称 |
| `args_matcher` | 参数匹配器 (None 表示不校验) |
| `order` | 在序列中的位置 |
| `required` | 是否必须出现 |
| `allow_retry` | 是否允许重试 |

**EvalWeights** — 默认评估权重：

| 维度 | 默认权重 |
|------|---------|
| output_correctness | 0.30 |
| tool_sequence_correctness | 0.25 |
| reasoning_quality | 0.20 |
| performance | 0.15 |
| error_recovery | 0.10 |

### 4.2 PostgreSQL 表结构

系统共 7 张核心表，通过 Alembic 管理迁移：

| 表名 | 职责 | 关键字段 |
|------|------|---------|
| `dataset_versions` | 数据集版本管理 | version_tag (UNIQUE), parent_version |
| `test_cases` | 测试用例存储 | input_messages (JSONB), expected_tool_calls (JSONB), tags (ARRAY + GIN) |
| `test_runs` | 测试批次记录 | agent_config (JSONB 快照), ab_group, optimization_id |
| `test_results` | 单条执行结果 | actual_output, actual_tool_calls (JSONB), full_trace (JSONB) |
| `evaluation_scores` | 评分明细 | dimension, score, weight, weighted_score, scoring_method, details (JSONB) |
| `optimizations` | 优化策略记录 | failure_analysis (JSONB), strategy_detail (JSONB), improvement_delta (JSONB) |
| `loop_control_log` | 闭环控制日志 | loop_session_id, iteration, aggregate_score, converged, safety_stopped |

表间关系：
```
dataset_versions ←── test_cases
                 ←── test_runs ←── test_results ←── evaluation_scores
                                ←── optimizations
                                ←── loop_control_log
```

### 4.3 数据集版本管理

`DatasetManager` 提供：
- `create_version(tag, description)` — 创建新版本
- `load_cases(version_tag)` — 加载指定版本的全部用例
- `fork_version(source_tag, new_tag, additional_cases)` — 从已有版本派生，追加新用例
- 优化层触发时自动 fork minor 版本并追加失败派生用例

### 4.4 用例自动生成

`CaseGenerator` 基于失败聚类调用 LLM 生成补充用例：
- 输入：失败聚类列表 (FailureCluster)
- 输出：针对每个失败模式生成 N 个变体测试用例
- 自动标记 `source="failure_derived"` 和 `tags=["failure:<category>"]`

## 5. 评估层详细设计

### 5.1 全链路 Trace 采集

`TraceCollectorCallback` 继承 LangChain 的 `BaseCallbackHandler`，零侵入挂载：

```
Agent 执行
  ├── on_llm_start/end     → ReasoningStep (tokens, latency, content, tool_calls_requested)
  ├── on_tool_start/end    → ToolCallRecord (name, input, output, duration)
  ├── on_tool_error        → ToolCallRecord (name, input, error)
  └── on_agent_finish      → final_output
```

输出 `AgentTrace` 对象，包含：
- `reasoning_steps: list[ReasoningStep]` — 每次 LLM 调用的完整记录
- `tool_calls: list[ToolCallRecord]` — 每次工具调用的完整记录
- `total_tokens / prompt_tokens / completion_tokens` — Token 统计
- `total_latency_ms` — 总延迟
- `final_output` — 最终输出
- `error` — 执行错误（如有）

### 5.2 五维评分体系

| 维度 | 权重 | 评分方式 | 核心逻辑 |
|------|------|---------|---------|
| **输出正确性** | 0.30 | 规则 + LLM 混合 | 规则：关键词/正则匹配 (0.4权重)；LLM：语义正确性判断 (0.6权重) |
| **工具调用序列** | 0.25 | 规则为主 | 覆盖率 (0.35) + 顺序对齐/LCS (0.25) + 参数匹配 (0.25) + 冗余惩罚 (0.15) |
| **推理质量** | 0.20 | LLM Judge | 逻辑连贯性 + 信息利用率 + 幻觉检测 + 推理效率 |
| **性能指标** | 0.15 | 纯规则 | 延迟/Token/工具调用次数 vs 阈值的比值，取平均 |
| **错误恢复** | 0.10 | 规则 + LLM 混合 | 规则：检测重试/换策略行为 (0.4)；LLM：恢复策略合理性 (0.6) |

评分器采用策略模式，所有评分器继承 `DimensionScorer` 基类，可灵活扩展新维度。

### 5.3 评估编排器

`EvaluationOrchestrator` 负责：
- `evaluate_single(agent, test_case)` — 执行单个用例 → 采集 Trace → 五维评分 → 加权总分
- `evaluate_batch(agent, test_cases, concurrency=5)` — 信号量控制并发，返回 `RunSummary`

`RunSummary` 包含：总用例数、通过数、失败数、总分、各维度平均分、逐用例结果。

### 5.4 回归检测

`RegressionDetector` 逐用例逐维度对比两次运行的分数：
- 阈值：delta < -0.05 视为回归
- 输出 `RegressionReport`，包含回归项列表和最严重回归

## 6. 优化层详细设计

### 6.1 失败模式分析

`FailureAnalyzer` 三步流程：

**Step 1 — LLM 分类**：对每个低分用例（总分 < 0.7），调用 LLM 分类到预定义类别：

| 失败类别 | 说明 |
|---------|------|
| `tool_selection_error` | 选错工具 |
| `tool_param_error` | 工具参数错误 |
| `missing_tool_call` | 遗漏必要工具调用 |
| `redundant_tool_call` | 多余工具调用 |
| `reasoning_error` | 推理逻辑错误 |
| `hallucination` | 幻觉 |
| `instruction_following` | 未遵循指令 |
| `context_loss` | 上下文丢失 |
| `error_handling` | 错误处理不当 |

**Step 2 — 聚类**：按分类标签聚类，生成 `FailureCluster`

**Step 3 — 摘要**：LLM 对每个聚类生成失败模式摘要和修复方向建议

### 6.2 策略生成

`StrategyGenerator` 调用 LLM 生成优化策略：

- **输入**：失败聚类 + 当前 Agent 配置（system prompt、工具列表、参数）+ 历史优化记录（避免重复）
- **输出**：`OptimizationStrategy`，包含：
  - `strategy_type`: "prompt" / "tool_config" / "system_param" / "composite"
  - `changes`: 具体修改列表，每项含 change_type、target、before、after、reason
  - `rationale`: 整体策略说明
  - `expected_improvement`: 预期各维度改进幅度
  - `risk_level`: "low" / "medium" / "high"

### 6.3 策略应用

`StrategyApplicator` 将策略应用到 Agent 配置的深拷贝上：

| change_type | 操作 |
|-------------|------|
| `prompt_modification` | 替换 system_prompt |
| `tool_description` | 修改指定工具的描述 |
| `tool_parameter` | 修改指定工具的默认参数 |
| `system_parameter` | 修改 temperature、max_tokens 等系统参数 |

## 7. 闭环控制器

### 7.1 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `target_score` | 0.85 | 目标总分，达到即收敛 |
| `max_iterations` | 10 | 最大迭代次数硬限制 |
| `min_improvement` | 0.01 | 最小改进幅度，低于此视为停滞 |
| `stagnation_patience` | 3 | 连续停滞轮数上限 |
| `regression_tolerance` | 0.05 | 回归容忍度 |
| `enable_ab_test` | true | 是否启用 A/B 测试验证 |
| `ab_test_ratio` | 0.3 | A/B 测试中 treatment 组比例 |

### 7.2 三重安全阀

1. **最大迭代次数**：硬限制，防止无限循环
2. **停滞检测**：连续 `stagnation_patience` 轮改进 < `min_improvement` → 自动停止
3. **回归保护**：优化后任何维度退化超过 `regression_tolerance` → 回滚到最佳配置

### 7.3 停止原因

| stop_reason | 含义 |
|-------------|------|
| `target_reached` | 目标分数达标，正常收敛 |
| `stagnation` | 连续多轮无改进 |
| `regression_with_stagnation` | 回归 + 停滞 |
| `max_iterations_reached` | 达到最大迭代次数 |
| `no_failures_to_optimize` | 无失败用例可优化 |
| `no_strategy_changes` | 策略生成器未产出有效变更 |

## 9. CLI 使用方式

```bash
# 初始化数据库
agent-eval init-db

# 创建数据集并导入用例
agent-eval create-dataset v1.0.0 --description "初始测试集" --from-file cases.json

# 仅评估（不优化）
agent-eval evaluate v1.0.0 --agent-module myapp.agent:factory

# 运行完整闭环优化
agent-eval run v1.0.0 \
    --agent-module myapp.agent:factory \
    --target-score 0.85 \
    --max-iterations 10 \
    --concurrency 5
```

### Agent Factory 接口约定

用户需要提供一个 Agent Factory，实现以下协议：

```python
class AgentFactory(Protocol):
    def get_config(self) -> dict[str, Any]:
        """返回当前 Agent 配置（system_prompt, tools, temperature 等）"""
        ...

    def create(self, config: dict[str, Any]) -> AgentProtocol:
        """根据配置创建 Agent 实例"""
        ...
```

## 13. 日志与可观测性

### 13.1 配置入口

`src/agent_eval/logging_config.py` 是唯一配置入口，由 `api/app.py` 顶部和 `cli.py` 的 `__main__` 各调用一次：

```python
from agent_eval.logging_config import setup_logging
setup_logging(settings.logging.level, settings.logging.format)
```

通过环境变量调节（前缀 `LOG_`）：

| 变量 | 默认 | 说明 |
|---|---|---|
| `LOG_LEVEL` | `INFO` | DEBUG / INFO / WARNING / ERROR |
| `LOG_FORMAT` | `plain` | `plain`（含 request_id 文本） / `json`（每行一条 JSON） |
| `LOG_DEBUG` | `false` | true 时把 traceback 暴露给前端响应；生产保持 false |
| `LOG_REQUEST_BODY` | `false` | true 时在 4xx/5xx 记请求体（≤4KB）；含敏感数据风险，谨慎打开 |

### 13.2 request_id 链路

`api/middleware.py` 的 `RequestContextMiddleware` 在每个请求入口：
1. 取请求头 `X-Request-ID`，没有就生成 `uuid4().hex[:16]`
2. 写入 `request.state.request_id` 与 `contextvars.ContextVar`
3. 响应头回写 `X-Request-ID: <id>`

`logging_config.RequestIdFilter` 把 contextvar 注入每条 log record，formatter 模板里 `[%(request_id)s]` 自动渲染——业务代码不需要每次手动传。

### 13.3 错误响应契约

全局 `Exception handler` 默认返回（向前端兼容 `detail` 字段）：

```json
{
  "detail": "Internal server error: ValueError",
  "request_id": "abc123def456"
}
```

`LOG_DEBUG=true` 时附加 `error_type / error_message / traceback`，仅供本地开发和受控环境。生产用 `request_id` 去 log 里 grep 完整 traceback。

### 13.4 日志字段命名约定

中间件输出的 access log 用以下 `key=value` 字段，便于 Grafana / Loki 检索：

| 字段 | 类型 | 含义 |
|---|---|---|
| `method` | string | HTTP 方法 |
| `path` | string | URL path（不含 query） |
| `query` | string | URL query string，无则 `-` |
| `client` | string | 客户端 IP，无则 `-` |
| `status` | int | HTTP 状态码 |
| `latency_ms` | float | 耗时毫秒 |

### 13.5 第三方 logger 压级

`logging_config.py` 把以下 logger 固定为 `WARNING`，避免 INFO 级别刷屏：

- `httpx`、`httpcore`（含 `http11`/`connection`）
- `langsmith`（含 `client`）
- `langgraph`
- `sqlalchemy.engine`、`asyncio`

需要单独 debug 某个第三方时，临时调用 `logging.getLogger("httpx").setLevel(logging.DEBUG)` 即可，不需要改全局配置。

### 13.6 测试侧

`setup_logging` 不使用 `force=True`，所有 logger 保持 `propagate=True`，pytest 的 `caplog` fixture 仍能捕获。`LoggingSettings` 字段全有默认值，不影响 `Settings()` 的现有初始化路径。
